# 도전 모드 설계 메모 (BE A 시현 · 회의용 초안)

> 목적: 4주차 예정 "도전 모드" 기능을 만들기 전에, 팀이 함께 정해야 할 것들을 정리한 제안서.
> 이 문서는 **결정이 아니라 제안**이며, 회의에서 방식을 골라 확정하는 용도.

---

## 1. 도전 모드란?

일반 면접과의 차이:

| | 일반 면접 (완성됨) | 도전 모드 (예정) |
|---|---|---|
| 질문 수 | 5개 (서로 다름) | **1개** |
| 답변 방식 | 각 질문에 1번씩 | **같은 질문에 5번 반복** |
| 목적 | 여러 질문 경험 | **한 답변을 다듬으며 실력 향상** |

핵심: "한 질문을 계속 고쳐 답하며 완성도를 높이는 연습"

---

## 2. 회의에서 정해야 할 것 (제일 중요)

### 결정 ①: 5번의 답변을 어떻게 평가할까?

**방식 A — 매번 평가**
- 답변할 때마다 점수·피드백 제공
- 장점: "점수가 점점 오르는 게 보임" → 성취감, 도전 모드 취지에 맞음
- 단점: Groq를 5번 호출 → 응답 대기 5번, 비용·시간 증가

**방식 B — 마지막만 평가**
- 5번 다듬은 뒤 최종 답변 1개만 평가
- 장점: Groq 1번만 호출 → 빠름
- 단점: 중간 과정의 "성장"이 안 보임

> 추천: **방식 A** (도전 모드의 재미는 "점점 나아지는 것"이라, 매번 평가가 취지에 맞음). 단, 속도가 걱정되면 B도 고려.

### 결정 ②: 5번이 같은 질문일까, 점점 바뀔까?

- **옵션 1**: 완전히 같은 질문 5번 (제일 단순)
- **옵션 2**: 이전 답변을 반영해 다음 질문이 심화됨 (구현 복잡, AI 부담 큼)

> 추천: **옵션 1** (같은 질문). MVP엔 단순한 게 안전. 옵션 2는 여유 있으면 확장.

### 결정 ③: 프론트 화면·흐름

- 도전 모드 진입 버튼은 어디에? (메인 화면?)
- 5번 답변을 한 화면에서 누적으로 보여줄지, 매번 새 화면인지?
- → **프론트 담당과 협의 필요** (현재 프론트 README에 도전 모드 화면 없음)

## 3. 백엔드 작업 범위 (BE A)

좋은 소식: **기존 코드를 대부분 재활용**할 수 있음.

### 필요한 DB 변경
- `interview_sessions`에 `mode` 컬럼 추가 (`'normal'` / `'challenge'` 구분)
  ```sql
  ALTER TABLE interview_sessions ADD COLUMN mode VARCHAR(20) DEFAULT 'normal';
  ```

### 필요한 API (방식 A 기준)
| API | 하는 일 | 재활용 |
|---|---|---|
| `POST /api/interview/challenge/start` | 질문 **1개** 생성 + 세션 생성(mode='challenge') | 기존 질문생성 로직 거의 그대로 (5개→1개만) |
| `POST /api/interview/challenge/answer` | 답변 저장 + 평가 (5번까지 반복 호출) | 기존 feedback 로직 그대로 |

→ 즉 **새로 짜는 게 아니라, 기존 질문생성·평가 함수를 살짝 변형**하는 수준.

### 작업량 예상
- DB 컬럼 추가: 5분
- challenge/start: 기존 `generateQuestions`에서 "5개→1개"만 바꾼 버전 (30분)
- challenge/answer: 기존 `evaluateAnswer` 거의 그대로 (20분)
- → **반나절 이내**. 단, 프론트 흐름 확정 후 진행해야 함.

## 5. BE A 의견

- 기존 코드 재활용도가 높아 **구현 자체는 부담 적음**.
- 다만 **프론트 화면이 없는 상태**라, 백엔드만 먼저 만들면 나중에 안 맞을 위험. → **프론트 흐름 확정 후 착수 권장.**
- 개인적으론 방식 A(매번 평가) + 옵션 1(같은 질문)이 "단순하면서 도전 모드다운" 조합이라고 봄.
